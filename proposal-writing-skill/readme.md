README.md — 方案生产流程说明（建议版）
📌 方法定位

本项目并非“方案写作工具”，而是用于支撑政府 / 公安信息化项目方案编制过程控制的一套结构化工作流。

核心目标是：

降低需求理解偏差

控制技术路线漂移

避免投资合理性风险

提升方案评审通过概率

方案的本质被视为：
决策与约束的表达载体，而非文本创作行为。

🧭 方案生产十步法（核心流程）

方案编制过程被拆解为以下十个阶段：

1、 用户需求输入（非结构化阶段）

来源包括但不限于：

会议纪要

用户口述

问题描述

厂商交流材料

特点：

信息可能不完整

技术语言可能混乱

不具备直接写方案条件

本阶段禁止直接进入方案写作。

2、需求澄清与修正（关键控制阶段）

目标：

去除歧义与冲突

剔除不可实施设想

明确系统边界

该阶段本质属于认知治理，而非技术设计。

3、 转换为 specs/需求.md（事实基线阶段）

所有需求必须固化为结构化文档：

specs/需求.md


该文件是后续全部决策与方案生成的唯一事实依据（Single Source of Truth）。

一旦确认，不应随意变更。

4、需求基线评审（合规校验阶段）

验证：

是否存在逻辑冲突

是否符合评审语境

是否具备实施与采购可行性

5、 技术方案空间推导与选择（决策核心阶段）

包含两个子阶段：

5A 技术路线空间识别
识别可行技术路径集合，而非直接推荐方案。

5B 技术路线裁决
基于风险、运维能力、投资合理性进行决策。

禁止以技术先进性作为主要判断依据。

6、 原厂询价 / 厂商测算（现实验证阶段）

目的：

验证市场可获得性

验证造价区间合理性

避免评审阶段价格质疑

7、 投资模型与整体报价构建（经济合理性阶段）

形成：

CAPEX 结构

设备与扩容模型

预算可解释路径

8、投资合理性评审（高风险阶段）

重点审查：

是否存在过度设计

是否符合财政投资逻辑

是否易被专家否决

9、 建设方案书写（表达阶段）

在前序阶段全部稳定后，方可进入文本生成：

不新增需求

不扩展能力

不改变决策逻辑

方案文本用于表达决策，而非探索决策。

10、 综合评审与一致性复核（终审阶段）

验证：

技术逻辑一致性

投资逻辑一致性

章节逻辑一致性

可答辩性

⚠ 核心原则

需求优先于方案

决策优先于写作

风险控制优先于技术偏好

投资合理性优先于功能扩展

🤖 与 Claude / Skill 的关系

Claude 与 Skill 体系用于：

执行结构化推理

控制章节生成逻辑

避免写作漂移

强制评审语境一致

模型不承担需求创造职责，
仅承担决策表达与规则执行职责。

✅ 使用建议

在任何方案生成行为之前，必须确保：

✔ specs/需求.md 已完成并评审通过
✔ 技术路线已完成裁决
✔ 投资模型无冲突

否则禁止进入方案生成阶段。





# Proposal Writing Skill

本 Skill 用于在 Claude Code 中生成与维护政府 / 公安 / 信息化类方案文档。

Skill 的职责是控制模型写作行为与输出结构，
而非提供内容建议或技术决策。

---

## 一、Skill 目录结构

proposal-writing-skill/
├── SKILL.md              # Skill 规则权威文件（模型行为与写作约束）
├── templates.md          # 方案章节模板（结构骨架）
├── tone.md               # 语气与风格约束（政府/公安写作语境）
├── forbidden.md          # 禁止项清单（禁止扩展、禁止重写等）
├── examples.md           # 使用示例（正确/错误prompt、章节示例）
├── specs/                # 方案“事实与约束”权威输入（如：需求.md）
│   └── 需求.md           # 项目需求与约束定义（方案级，Single Source of Truth）
├── drafts/               # 方案草稿区（中间态章节/临时内容）
│   ├── chart/            # 草稿图/推理图（可选：mermaid/html等）
│   └── *.md              # 各章节草稿（按章节拆分）
└── output/               # 正式交付区（最终方案与可交付资产）
    ├── chart/            # 正式图件（拓扑/架构/部署等；可为html或导出的png/svg/pdf）
    └── *.md / *.docx     # 最终方案文本（按你的交付形态）


SKILL.md 是唯一规则权威文件。
其余文件均为规则支撑组件。

---

## 二、Skill 适用场景

适用于：

✔ 建设方案生成  
✔ 章节补充  
✔ 行文风格校正  
✔ 结构化写作任务  

---

## 三、Skill 调用方式

使用 Skill 时，提示词只描述任务。

示例：

按 proposal-writing Skill，
生成《项目理解》章节。

按 proposal-writing Skill，
对以下内容进行章节增强：

按 proposal-writing Skill，
生成本项目建设方案结构。

---

## 四、Skill 使用原则

- 不在提示词中重复描述写作规则
- 不在提示词中描述语气与风格要求
- 规则统一由 SKILL.md 控制

---

## 五、Skill 设计目标

- 稳定输出
- 结构一致
- 避免模型自由发挥

视觉风格要求：
