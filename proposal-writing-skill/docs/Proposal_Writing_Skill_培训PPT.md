## P03｜方法定位

### Proposal Writing Skill 不是什么

- 不是"AI 写方案"工具
- 不是技术顾问
- 不提供技术建议
- 不参与需求判断

### Proposal Writing Skill 是什么

- 政府 / 公安信息化项目**方案编制过程控制体系**
- 控制模型写作行为与输出结构的执行工具
- 强制模型服从权威文件、禁止自由发挥

### 核心定义

**方案的本质 = 决策与约束的表达载体，而非文本创作行为。**

> Skill 负责"怎么说"，不负责"说什么"。
> "说什么"由 specs/ 权威文件决定。

## P04｜核心原则

### 四条不可违反的优先级原则

| 优先级 | 原则 |
|--------|------|
| 1 | 需求优先于方案 |
| 2 | 决策优先于写作 |
| 3 | 风险控制优先于技术偏好 |
| 4 | 投资合理性优先于功能扩展 |

### 工程意义

- **需求优先于方案**：没有固化需求，不得生成方案
- **决策优先于写作**：未完成技术路线裁决，不得进入章节写作
- **风险控制优先于技术偏好**：不以技术先进性作为方案选型依据
- **投资合理性优先于功能扩展**：方案能力边界由投资模型决定，不得超出

> 这四条原则是评审阶段最常见的被质疑点，
> 也是最需要在编制阶段提前控制的风险维度。

## P05｜整体文件体系架构

### Skill 文件层级结构

```
proposal-writing-skill/
├── SKILL.md                    ← 唯一规则权威文件
├── specs/                      ← 最高权威层（事实与约束）
│   ├── 需求.md                 ← 业务事实 SSoT
│   ├── technical_baseline.md   ← 技术路线裁决文件
│   ├── architecture_scope.md   ← 架构范围约束
│   ├── investment_schema.md    ← 资源配置与投资模型
│   ├── chart_registry.md       ← 图件注册清单
│   └── document_conventions.md ← 文档排版规范
├── drafts/                     ← 草稿区（中间态）
│   └── chart/                  ← 推理图（Mermaid）
└── output/                     ← 正式交付区
    ├── chart/                  ← 正式图件（HTML→PNG）
    ├── screenshot/             ← 现网截图（独立管理）
    └── *.md / *.docx           ← 最终方案文本
```

### 关键规则

- `specs/` 内容一旦确认，**不得随意变更**
- SKILL.md 是唯一行为规则文件，其余文件是规则支撑组件
- 正式图件不得从 `drafts/` 直接复制到 `output/`

## P06｜权威优先级

### 五个层级的冲突裁决规则

| 优先级 | 文件 / 层级 | 说明 |
|--------|-------------|------|
| 1（最高） | `specs/` 目录 | 业务事实与约束，所有决策的根基 |
| 2 | `drafts/` | 中间草稿，禁止整体重写 |
| 3 | `output/` | 正式交付物 |
| 4 | `PROPOSAL_PLAN.md` | 可丢弃的状态跟踪文件 |
| 5（最低） | 模型推理 / 经验判断 | **禁止作为事实来源** |

### specs/ 内部的冲突裁决

当 `specs/需求.md` 与 `specs/investment_schema.md` 存在资源规模冲突时：

- **investment_schema.md 优先**（资源配置权属于投资模型文件）
- 需求.md 仅定义业务与功能边界，**不裁决资源模型**

> 任何冲突，服从更高权威。不得以正文描述推翻 specs/ 文件。

## P07｜方案生产十步法概览

### 四个阶段、十个步骤

```
[需求阶段]          [决策阶段]          [验证阶段]          [输出阶段]
  Step 1              Step 4              Step 6              Step 9
用户需求输入        需求基线评审        原厂询价测算        建设方案书写
  Step 2              Step 5A             Step 7              Step 10
需求澄清与修正      技术路线识别        投资模型构建       综合评审复核
  Step 3              Step 5B             Step 8
固化为需求.md       技术路线裁决       投资合理性评审
```

### 核心控制逻辑

- **禁止跳步**：任何阶段未完成，不得进入下一阶段
- **禁止逆序**：已固化的文件不得被后续阶段推翻
- **禁止并行**：需求、决策、验证三阶段必须顺序完成

> Step 9（方案书写）是整个流程中最后进入的环节，
> 不是最先开始的环节。

## P08｜需求阶段详解（Step 1–3）

### Step 1｜用户需求输入（非结构化阶段）

来源类型：会议纪要 / 用户口述 / 问题描述 / 厂商交流材料

**本阶段特点：**
- 信息可能不完整
- 技术语言可能混乱或矛盾
- 不具备直接写方案的条件

**禁止行为：** 在 Step 1 完成前进入方案写作

---

### Step 2｜需求澄清与修正（认知治理阶段）

**目标：**
- 去除歧义与冲突
- 剔除不可实施设想
- 明确系统边界

> 本阶段属于**认知治理**，而非技术设计。
> 输出的是"确认过的事实"，不是"设计方案"。

---

### Step 3｜固化为 specs/需求.md（事实基线阶段）

**强制要求：**
- 必须包含"**现网设备清单**"章节 → 若缺失，**禁止进入方案生成**
- 现网结构表达仅可来源于此文件
- 文件一旦确认，不应随意变更

> `specs/需求.md` 是后续全部决策与方案生成的
> **唯一事实依据（Single Source of Truth）**。

---

---

## P09｜决策阶段详解（Step 4–5B）

### Step 4｜需求基线评审（合规校验）

**验证内容：**
- 是否存在逻辑冲突
- 是否符合评审语境（财政、招采合规）
- 是否具备实施与采购可行性

**产出：** 通过评审的 `specs/需求.md`（签字或确认版本）

---

### Step 5A｜技术路线空间识别

**目标：** 识别**可行技术路径集合**，而非直接推荐方案

- 列出候选路线及其约束条件
- 分析各路线的风险与运维成本
- 不对外发布此阶段内容

---

### Step 5B｜技术路线裁决（决策核心）

**裁决依据：**
- 风险可控性
- 运维能力匹配度
- 投资合理性

**禁止依据：**
- 技术先进性
- 厂商推荐
- 模型默认倾向

> 裁决结果写入 `specs/technical_baseline.md` 并**冻结**。
> 冻结后，任何方案内容不得偏离此基线。

---

---

## P10｜验证阶段详解（Step 6–8）

### Step 6｜原厂询价 / 厂商测算（现实验证）

**目的：**
- 验证设备市场可获得性
- 验证造价区间合理性
- 避免评审阶段出现价格质疑

**产出：** 厂商报价单（归档至 `specs/` 或附件）

---

### Step 7｜投资模型与整体报价构建

**形成内容：**
- CAPEX 结构（软硬件 / 集成 / 服务费分类）
- 设备与扩容模型
- 预算可解释路径（每项有依据）

**产出：** `specs/investment_schema.md`（金额冻结）

---

### Step 8｜投资合理性评审（高风险阶段）

**重点审查：**
- 是否存在过度设计（配置超出业务需求）
- 是否符合财政投资逻辑
- 是否易被专家否决（单价偏高、数量不合理）

> Step 8 通过后，投资模型不再调整。
> 任何后续章节不得扩展 `investment_schema.md` 未定义的能力。

---

---

## P11｜输出阶段详解（Step 9–10）

### Step 9｜建设方案书写（表达阶段）

**进入条件（全部满足方可开始）：**

- [ ] `specs/需求.md` 已完成并通过评审
- [ ] `specs/technical_baseline.md` 已冻结
- [ ] `specs/investment_schema.md` 已确认无冲突

**写作行为约束：**
- 不新增需求
- 不扩展未裁决能力
- 不改变既定决策逻辑

> **方案文本用于表达决策，而非探索决策。**

---

### Step 10｜综合评审与一致性复核（终审）

**验证维度：**
- 技术逻辑一致性（章节内容 ↔ technical_baseline）
- 投资逻辑一致性（预算数字 ↔ investment_schema）
- 章节结构一致性（各章之间不矛盾）
- 可答辩性（专家提问是否有据可查）

> 终审后方可进入交付与发布流程。

## P12｜技术路线冻结机制

### 冻结的含义

技术路线一旦写入 `specs/technical_baseline.md`，即视为**工程裁决结果**：

- 方案正文不得偏离基线文件
- 图件与部署结构不得引入基线外节点
- 章节描述不得进行替代路线比较
- 不得在方案中引入演进性技术变体

### 当前技术路线基线（示例）

| 层级 | 裁决结论 |
|------|---------|
| 计算架构 | 国产化 x86 架构服务器体系 |
| 操作系统 | 国产服务器操作系统 |
| 数据库 | 国产数据库管理系统（达梦等） |
| 存储模型 | NAS 统一文件存储资源池 |
| 设备层定位 | 数据采集接入层，不承担计算能力 |

### 明确排除的技术模型

- 分布式对象存储集群架构
- FC-SAN 集中式存储
- 云存储 / 云资源池模型
- 混合异构存储架构

> 以上排除项不属于"未来可选"，属于**本项目既定范围外**。
> 方案中不得出现，图件中不得表达。

## P13｜投资控制逻辑

### 投资模型的法律地位

`specs/investment_schema.md` 是本项目资源配置与预算的**唯一合法来源**：

- 服务器数量 / 设备配置参数 → 来源于此文件
- 存储容量规模 / 角色划分数量 → 来源于此文件
- 预算设备清单 / 部署结构 → 来源于此文件

### 禁止模型（人工 / AI）自行进行的操作

| 禁止行为 | 风险说明 |
|----------|---------|
| 自行计算服务器数量 | 与清单冲突，引发评审质疑 |
| 自行修改配置规模 | 破坏预算平衡 |
| 自行扩展硬件类别 | 新增未批准采购项 |
| 基于经验推导容量 | 无合同依据，无法答辩 |
| 修改总计金额 | 直接违反财政合规要求 |

### 计算约束（不可违反）

```
各分项金额之和 = 总计
若计算冲突 → 以总计为最高约束
不允许因模型推理调整预算结构
```

> 单价是**政策合规参数**，不是估算参数，不得基于市场均值推理生成。

## P14｜图件体系分类

### 图件的定位

图件属于**技术决策表达资产**，不是自由内容生成资产：

- 用于表达既定技术决策
- 用于解释系统结构关系
- 用于支撑方案评审理解

### 四类图件

| 类别 | 名称 | 启用条件 | 事实来源 |
|------|------|---------|---------|
| A类 | 架构决策类（**强制**） | 对应章节触发 | investment_schema / 需求.md |
| B类 | 实施与流程类 | 出现实施进度 / 切换策略 | 已裁决流程 |
| C类 | 投标增强类 | 仅投标文件自动启用 | — |
| D类 | UI 表达类 | 功能需求分析章节 | output/screenshot/ |

### A 类图件清单（强制生成）

| 编号 | 图件名称 | 输出路径 |
|------|---------|---------|
| A-01 | 系统总体架构图 | output/chart/system_architecture.html |
| A-02 | 网络拓扑图 | output/chart/network_topology.html |
| A-03 | 数据流转图 | output/chart/data_flow.html |
| A-04 | 部署结构图 | output/chart/deployment_structure.html |
| A-05 | 服务器角色划分图 | output/chart/server_role_structure.html |
| A-06 | 现网结构图 | output/chart/current_architecture.html |

> A-06 现网结构图：**事实来源唯一为 `specs/需求.md`**，
> 不得从 `investment_schema.md` 或正文推导。

## P15｜图件生成纪律

### 格式纪律（不可违反）

| 格式 | 用途 | 允许 |
|------|------|------|
| HTML | 正式图件源格式 | ✔ 唯一合法源格式 |
| PNG / SVG | 派生交付格式 | ✔ 必须基于 HTML 导出 |
| Mermaid | 推理草图 | ✔ 仅限 drafts/chart/ |
| ASCII 图 | — | **✗ 禁止** |
| 直接生成 PNG | — | **✗ 禁止** |

**PNG 分辨率要求：≥ 1920×1080**

### 图件生成执行顺序（强制）

```
① 完成正文结构生成
② 根据 chart_registry.md 判断触发条件
③ 生成 drafts/chart/ 推理图
④ 校验一致性（A-现状类 ↔ 需求.md；A-建设类 ↔ investment_schema）
⑤ 生成 output/chart/ 正式 HTML 图件
⑥ 导出 PNG
⑦ 正文引用图件
```

**禁止：正文未完成前提前生成正式图件。**

### 图件冲突裁决优先级

```
architecture_scope > technical_baseline > investment_schema > chart_registry > 正文描述
```

## P16｜写作行为约束

### 六条强制写作纪律

| 编号 | 约束规则 |
|------|---------|
| 1 | 不得整体重写既有章节 |
| 2 | 不得改变章节标题与编号结构 |
| 3 | 若章节已存在，仅允许增量补充 |
| 4 | 禁止删除用户未明确要求删除的内容 |
| 5 | 禁止扩展用户未提出的系统能力 |
| 6 | 禁止解释写作理由或模型决策过程 |

### 章节风格要求

| 允许 | 禁止 |
|------|------|
| 功能性章节 | 论文式结构 |
| 直接服务项目建设逻辑 | 背景化散文写法 |
| 客观中性表达 | 营销 / 宣传 / 产品介绍风格 |
| 结构清晰、可被摘录 | 情绪化表达 |
| 方案级抽象描述 | 工程设计书风格、算法细节 |

### 信息密度原则

- 删除无功能语句
- 优先结构清晰而非修辞
- 每段必须承担明确文档功能
- 禁止冗余解释

## P17｜常见违规示例对比（上）

### 场景一：需求未固化，直接进入写作

| 违规做法 | 正确做法 |
|---------|---------|
| 收到用户口述后，直接让 AI 生成建设方案 | 先将需求澄清、固化为 `specs/需求.md`，通过评审后再生成 |
| 方案内容来自会议讨论记录，未形成结构化文件 | 需求必须以文件形式固化，不得以"默认共识"替代文档 |

---

### 场景二：技术路线未冻结，章节出现多种方案对比

| 违规做法 | 正确做法 |
|---------|---------|
| 方案正文中写"可采用 A 方案或 B 方案" | 技术路线已裁决，正文只写裁决结果 |
| 章节中出现"也可考虑对象存储" | 对象存储已被明确排除，不得出现在方案中 |
| 以"技术更先进"为由推荐非基线技术 | 先进性不是裁决依据，合规性与可控性才是 |

---

### 场景三：投资数字与 investment_schema 不一致

| 违规做法 | 正确做法 |
|---------|---------|
| 正文写"共配置 3 台应用服务器" | 服务器数量必须来自 investment_schema，不得自行推导 |
| 根据存储容量需求反推硬盘数量 | 硬盘数量以 investment_schema 清单为准，不得计算 |
| 预算总计与各分项之和不等 | 各分类小计之和必须严格等于总计，不允许误差 |

## P18｜常见违规示例对比（下）

### 场景四：图件越权

| 违规做法 | 正确做法 |
|---------|---------|
| 拓扑图中出现 investment_schema 未定义的服务器 | 图件节点必须来自 investment_schema，不得新增 |
| 现网结构图参考了 investment_schema 的建设规划 | 现网结构图唯一来源是 `specs/需求.md` |
| 图件中出现完整 IP 地址 | 拓扑图、部署图不得出现完整 IP 地址 |
| 直接将 Mermaid 草图作为正式图件交付 | 正式图件必须为 HTML 格式，PNG 为派生交付格式 |

---

### 场景五：写作行为越权

| 违规做法 | 正确做法 |
|---------|---------|
| 在补充章节时重写了原有段落 | 既有章节只允许增量补充，不得整体重写 |
| 方案中扩展了"未来可扩展为 AI 分析平台" | 仅表达已裁决能力，不引入未来扩展描述 |
| 章节末尾加入"建议后续……" | 禁止建议性语气和未来能力设想 |
| 写作中解释了"此处这样写是因为……" | 禁止在方案中输出模型决策说明 |

---

### 场景六：现网与建设层混用

| 违规做法 | 正确做法 |
|---------|---------|
| 现状分析章节引用了建设后的设备规模 | 现状描述仅来自 `specs/需求.md` |
| 建设方案章节引用了现网存在的问题作为配置依据 | 建设配置来自 investment_schema，不以现网问题推导规模 |

## P19｜提交前检查清单

### 在提交或发布方案前，必须逐项确认

**一、事实层（specs/ 完整性）**

- [ ] `specs/需求.md` 已完成，包含"现网设备清单"章节
- [ ] `specs/technical_baseline.md` 已冻结，状态为"已裁决"
- [ ] `specs/investment_schema.md` 已确认，总计 = 各分项之和
- [ ] `specs/chart_registry.md` 已确认本项目适用图件类别

**二、技术路线一致性**

- [ ] 方案正文未出现基线排除的技术模型（对象存储 / 云架构 / 灾备中心等）
- [ ] 方案正文未进行技术路线对比或替代方案描述
- [ ] 图件中无未定义节点，无 IP 地址

**三、投资合理性**

- [ ] 正文中设备数量与 investment_schema 一致
- [ ] 预算章节金额与 investment_schema 完全一致
- [ ] 无过度设计描述（配置超出业务需求的解释）

**四、图件体系**

- [ ] A 类强制图件已全部生成（A-01 至 A-06）
- [ ] 正式图件均为 HTML 源格式，存放于 output/chart/
- [ ] PNG 均为 HTML 导出，分辨率 ≥ 1920×1080
- [ ] 现网截图存放于 output/screenshot/，独立管理

**五、写作合规性**

- [ ] 无整体重写既有章节行为
- [ ] 无章节标题或编号变更
- [ ] 无营销 / 宣传 / 论文式语言

**六、可答辩性**

- [ ] 每项技术选择有 specs/ 文件支撑

- [ ] 章节结构无内部矛盾

> 以上任何一项不满足，禁止进入交付流程。

## P20｜总结

### Proposal Writing Skill 的核心工程价值

| 控制维度 | 解决的问题 |
|---------|-----------|
| 需求固化机制 | 避免口头需求带来的理解偏差 |
| 技术路线冻结 | 避免方案在写作中漂移 |
| 投资模型权威 | 避免评审阶段的价格质疑 |
| 图件注册控制 | 避免图件越权表达未批准内容 |
| 写作行为约束 | 避免模型自由发挥破坏方案一致性 |

### 记住三个"禁止"

1. **禁止在需求未固化前进入写作**
2. **禁止在技术路线未冻结前生成图件与正文**
3. **禁止模型自行推导投资数字与设备配置**

### 记住一个"定义"

> **方案 = 决策与约束的表达载体。**
>
> 写方案不是在创作，而是在把"已经确定的事情"
> 用符合评审语境的方式说清楚。
