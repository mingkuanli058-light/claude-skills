# PROPOSAL WRITING SKILL

## 权威优先级（不可违反）

1. specs/            → 最高权威，定义业务与边界
2. drafts/           → 中间草稿，禁止整体重写
3. output/           → 正式交付物
4. PROPOSAL_PLAN.md  → 可丢弃状态文件

若发生冲突，必须服从更高权威。

---


## 技术路线控制规则（最高优先级）

technical_baseline.md 为技术路线唯一裁决依据。

- 若 technical_baseline.md 与任何正文内容冲突，必须服从 technical_baseline.md
- 不得在方案生成过程中改变或扩展基线技术路线
- 技术架构、图件、部署结构均不得偏离基线文件
- 图件生成必须服从 architecture_scope.md
- 技术描述必须保持方案级抽象
- 不进入实现细节、算法细节或产品实现机制层
- 禁止出现工程设计书风格内容


---

## 写作行为约束（强制规则）

- 不得整体重写既有章节
- 不得改变章节标题与编号结构
- 若章节已存在，仅允许增量补充
- 禁止删除用户未明确要求删除的内容
- 禁止扩展用户未提出的系统能力
- 禁止解释写作理由或模型决策过程

---

## 章节生成规则

- 章节必须为“功能性章节”，不得使用论文式结构
- 内容必须直接服务项目建设逻辑
- 禁止背景化散文写法
- 禁止营销 / 宣传 / 产品介绍风格
- 段落必须可被审查与摘录

---

## 语气与风格锁定

- 使用政府 / 公安 / 信息化建设语境
- 语气客观、中性、非宣传
- 不得使用情绪化表达
- 不得使用模型自我描述
- 不得出现“建议性闲聊语气”

---

## 信息密度要求

- 删除无功能语句
- 优先结构清晰而非修辞
- 每段必须承担明确文档功能
- 禁止冗余解释

---

## 文件操作纪律

- 写作前必须假定文件已存在
- 若需修改 drafts/，只做局部补充
- 不得重建用户已有结构
- 不得输出与任务无关章节

---

## 默认目标

生成内容必须可直接用于：

✔ 项目汇报  
✔ 立项论证  
✔ 招投标材料  
✔ 技术审查  
✔ 内部审批  


---

## 投资预算生成规则（强制）

投资预算相关内容必须服从：

- specs/investment_schema.md
- specs/technical_baseline.md
- specs/需求.md
- 单价属于政策合规参数，而非估算参数
- 不得基于市场均值、经验值或模型推理生成


禁止模型自行设计预算结构或计算规则。


---

## 图件与技术示意生成规则（强制）

### 一、图件地位

- 图件属于正式方案交付物组成部分  
- 图件用于表达系统结构、部署关系与技术决策  
- 图件不得作为装饰性内容生成  
- 禁止生成与项目无关的示意图  

---

### 二、图件触发条件

当方案内容涉及以下任一情形时，必须生成对应图件：

- 涉及网络结构、设备连接、链路关系 → 必须生成 **网络拓扑图**
- 涉及系统层级、组件构成、模块关系 → 必须生成 **系统架构图**
- 涉及数据流转、数据存储、交互路径 → 必须生成 **数据流转图**
- 涉及系统部署位置、节点分布、中心结构 → 必须生成 **部署结构图**

不得省略，不得替代。

---

### 三、图件生成位置纪律

- 推理或草稿图件 → drafts/chart/
- 正式交付图件 → output/chart/

禁止：

- 将 drafts 图件直接作为交付物
- 在 output 中生成未经推理的临时图
- 在正文中使用 ASCII 图或伪图替代专业图件

---

### 四、图件一致性约束

所有图件必须：

- 与 specs/需求.md 严格一致
- 不得引入 specs 未定义的系统元素
- 不得扩展未裁决的技术组件
- 不得隐含新增系统能力
- 命名符合政府 / 公安方案语境
- 图件不得基于常识、行业惯例或模型推理补充 specs 未明确的元素
- 正文技术描述不得与既有图件结构发生冲突
- 若正文与图件存在不一致，必须以图件表达结构为准

禁止：

- 自行设计新系统结构
- 引入未说明的第三方系统
- 使用厂商宣传式架构表达

---

### 五、图件表达原则

图件仅用于：

- 表达既定技术决策
- 解释系统结构关系
- 支撑方案评审理解

图件不得承担：

- 技术探索功能
- 方案创新展示功能
- 产品宣传功能

---

### 六、图件格式纪律

图件源格式唯一规则（不可违反）

- 正式图件的源格式必须为 HTML
- PNG / SVG 仅允许作为派生交付格式
- 不得将 PNG / SVG 作为源图件生成

Mermaid 仅允许用于 drafts 阶段推理图件。

禁止：

- 在未指定格式时自行选择复杂绘图方案
- 在正文 Markdown 中嵌入图形脚本
- 输出不可直接交付的中间格式

---

### 七、图件生成行为限制

- 图件生成不得改变方案文本结构
- 图件生成不得触发章节重写
- 图件生成不得扩展系统能力描述
- 图件仅表达，不得推导新设计

---

### 八、图件属性定义

图件属于 **技术决策表达资产**，  
而非 **内容生成资产**。

---

### 九、图件输出与交付格式规则（强制）

1. 所有正式图件必须首先生成 HTML 版本  
   输出路径：output/chart/

2. HTML 图件属于权威源图件，不得跳过生成

3. PNG / SVG 图片属于派生交付资产  
   必须基于 HTML 图件生成

4. 图片生成方式：

   - 通过 HTML 渲染后导出
   - 或通过截图生成

5. 禁止：

   - 模型直接生成 PNG 作为源图
   - 跳过 HTML 图件直接生成图片
   - 图文内容不一致

6. 图片用途限定：

   - Word / PDF 嵌入
   - 招投标文件
   - 静态汇报材料

7. HTML 图件用途限定：

   - 技术讲解
   - 汇报演示
   - 结构审查

---

### 十、图件分辨率与比例纪律（强制）

- PNG 图片分辨率不得低于 1920×1080
- 禁止输出低清晰度图件
- 图件比例必须适用于 Word / PPT 嵌入
- 禁止生成不可读或文字模糊的图件
