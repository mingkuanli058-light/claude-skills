# Proposal Writing Skill

用于在 Claude Code 中生成与维护政府 / 公安 / 信息化类建设方案的结构化工作流。

**Skill 的职责是控制模型写作行为与输出结构，不承担内容建议或技术决策职责。**

---

## 一、方法定位

本 Skill 不是"方案写作工具"，而是用于政府 / 公安信息化项目方案编制过程控制的结构化执行体系。

核心目标：

- 降低需求理解偏差
- 控制技术路线漂移
- 避免投资合理性风险
- 提升方案评审通过概率

方案的本质被定义为：**决策与约束的表达载体，而非文本创作行为。**

## 二、目录结构

```
proposal-writing-skill/
├── SKILL.md                    # Skill 规则权威文件（模型行为与写作约束）
├── readme.md                   # 本文件
├── specs/                      # 方案"事实与约束"权威输入层（最高权威）
│   ├── 需求.md                 # 项目需求与约束定义（Single Source of Truth）
│   ├── technical_baseline.md   # 技术路线唯一裁决依据
│   ├── architecture_scope.md   # 图件与架构范围约束
│   ├── investment_schema.md    # 资源配置与投资模型（资源裁决权威）
│   ├── chart_registry.md       # 图件注册与触发控制清单
│   └── document_conventions.md # 文档排版规范
├── drafts/                     # 方案草稿区（中间态，禁止整体重写）
│   └── chart/                  # 草稿推理图（Mermaid / HTML）
├── output/                     # 正式交付区
│   ├── chart/                  # 正式图件（HTML 源图 + PNG 派生）
│   ├── screenshot/             # 现网系统截图（独立管理）
│   └── *.md / *.docx           # 最终方案文本
└── docs/                       # 工程参考文档（速查卡、流程图等）
```

**权威优先级（不可违反）：**

```
specs/ > drafts/ > output/ > PROPOSAL_PLAN.md
```

若发生冲突，必须服从更高权威。

## 三、方案生产十步法

方案编制过程被拆解为以下十个阶段，**禁止跳步或逆序执行**。

| 阶段 | 名称 | 关键控制点 |
|------|------|-----------|
| 1 | 用户需求输入 | 非结构化阶段，禁止直接进入写作 |
| 2 | 需求澄清与修正 | 去歧义、明确边界，属于认知治理 |
| 3 | 转换为 specs/需求.md | 固化为事实基线，唯一事实来源 |
| 4 | 需求基线评审 | 逻辑冲突校验、可行性验证 |
| 5A | 技术路线空间识别 | 识别可行路径集合，不直接推荐 |
| 5B | 技术路线裁决 | 基于风险、运维、投资进行决策 |
| 6 | 原厂询价 / 测算 | 验证市场可获得性与造价合理性 |
| 7 | 投资模型构建 | 形成 CAPEX 结构与预算可解释路径 |
| 8 | 投资合理性评审 | 审查过度设计、财政投资逻辑 |
| 9 | 建设方案书写 | 前序全部稳定后方可进入文本生成 |
| 10 | 综合评审与一致性复核 | 技术 / 投资 / 章节逻辑终审 |

> **禁止以技术先进性作为技术路线的主要判断依据。**

---

## 四、核心原则

- 需求优先于方案
- 决策优先于写作
- 风险控制优先于技术偏好
- 投资合理性优先于功能扩展

| 场景 | 是否适用 |

## 六、核心提示词

### 方案生成

```
按 proposal-writing Skill，
严格依据以下权威文件：
- specs/需求.md（必须包含"现网设备清单"章节，若缺失 → 终止生成）
- specs/technical_baseline.md
- specs/architecture_scope.md
- specs/investment_schema.md

按 templates.md 章节结构，生成本项目建设方案。
```

### 方案扩写

```
按照 specs/document_conventions.md 的要求，
对 output/建设方案.md 进行扩写。
```

### 转 Word 交付

```
将 output/建设方案.md 转为 Word，
使用政府信息化方案排版规范，
自动插入 output/chart/ 中全部图件。
```

### 生成政府汇报 PPT

```
按 proposal-writing Skill，
严格依据以下权威文件：
- output/建设方案.md
- specs/presentation_rules.md

生成政府汇报 PPT。

视觉风格要求：
- 整体保持政务汇报气质，风格稳重、克制、专业
- 类指挥中心 / 综合管控平台视觉
- 深色或低饱和背景
- 禁止互联网产品风、商业路演风、厂商宣传风
```

## 七、图件体系说明

图件属于**技术决策表达资产**，不属于自由内容生成资产。

图件分为四类，触发与校验规则见 [specs/chart_registry.md](specs/chart_registry.md)：

| 类别 | 说明 | 事实来源 |
|------|------|---------|
| A类（强制）| 架构决策类（拓扑、部署、数据流等） | investment_schema / 需求.md |
| B类 | 实施与流程类（甘特图、切换流程等） | 已裁决流程 |
| C类 | 投标增强类（仅投标文件启用） | — |
| D类 | UI表达类（界面示意，独立存放） | output/screenshot/ |

**图件格式纪律：**

- 正式图件源格式必须为 **HTML**，输出至 `output/chart/`
- PNG / SVG 仅允许作为派生交付格式（分辨率 ≥ 1920×1080）
- Mermaid 仅允许用于 `drafts/chart/` 阶段推理图
- 禁止使用 ASCII 图替代正式图件

**图件冲突裁决优先级：**

```
architecture_scope > technical_baseline > investment_schema > chart_registry > 正文描述
```

## 八、使用前置条件

进入方案生成阶段前，必须确认：

- [ ] `specs/需求.md` 已完成并包含"现网设备清单"章节
- [ ] 技术路线裁决完成（`specs/technical_baseline.md` 已确认）
- [ ] 投资模型无冲突（`specs/investment_schema.md` 已确认）

**以上任一条件不满足，禁止进入方案生成阶段。**

---

## 九、写作行为约束（强制）

- 不得整体重写既有章节
- 不得改变章节标题与编号结构
- 若章节已存在，仅允许增量补充
- 禁止扩展用户未提出的系统能力
- 禁止解释写作理由或模型决策过程
- 章节必须为"功能性章节"，禁止论文式、散文式、宣传式写法
