# PROPOSAL WRITING SKILL

需求.md 属于现网事实层资产，
investment_schema.md 属于建设规划层资产。

两者不得混用。

## 图件风格变量

CHART_STYLE = review | presentation | tender
默认：presentation

UI_STYLE = wireframe | neutral | enhanced
默认：enhanced

文档类型为“投标文件”时：
CHART_STYLE 自动切换为 tender

文档类型为“内部材料”时：
CHART_STYLE 自动切换为 review

---


## 权威优先级（不可违反）

1. specs/            → 最高权威，定义业务与边界
2. drafts/           → 中间草稿，禁止整体重写
3. output/           → 正式交付物
4. PROPOSAL_PLAN.md  → 可丢弃状态文件

若发生冲突，必须服从更高权威。

---


## 技术路线控制规则（最高优先级）

technical_baseline.md 为技术路线唯一裁决依据。

- 若 technical_baseline.md 与任何正文内容冲突，必须服从 technical_baseline.md
- 不得在方案生成过程中改变或扩展基线技术路线
- 技术架构、图件、部署结构均不得偏离基线文件
- 图件生成必须服从 architecture_scope.md
- 技术描述必须保持方案级抽象
- 不进入实现细节、算法细节或产品实现机制层
- 禁止出现工程设计书风格内容

## 图件注册与执行机制（强制）

图件生成必须服从：

specs/chart_registry.md

图件触发逻辑不得依赖关键词自由判断，
必须依据 chart_registry.md 中定义的：

- 图件类别
- 触发条件
- 输出路径
- 风格变量
- 是否强制生成

若本文件图件规则与 chart_registry.md 存在冲突：

必须服从 chart_registry.md。

## 图件引擎 v4（强制使用）

图件生成架构：

```
AI 生成 chart_data.json
        ↓
chart/renderer.html 自动布局 + 渲染
        ↓
PNG 导出
```

### AI 只输出 JSON 结构

JSON 规范：chart/chart_schema.md

### AI 禁止输出

- HTML 代码
- SVG 代码
- x / y 坐标

### 渲染引擎

- 渲染器：chart/renderer.html
- 布局引擎：chart/layout_engine.js
- 支持类型：见 chart/chart_schema.md 中 chart_type 字段定义

### JSON 保存路径

```
output/chart_spec/<chart_type>.json
```

### 渲染步骤

1. 将 JSON 内容替换 renderer.html 中的 `{{chart_data}}`
2. 浏览器打开 renderer.html
3. 点击"导出 PNG"按钮

或使用：

```
output/html2png.py
```

## 写作行为约束（强制规则）

- 不得整体重写既有章节
- 不得改变章节标题与编号结构
- 若章节已存在，仅允许增量补充
- 禁止删除用户未明确要求删除的内容
- 禁止扩展用户未提出的系统能力
- 禁止解释写作理由或模型决策过程

---

## 章节生成规则

- 章节必须为“功能性章节”，不得使用论文式结构
- 内容必须直接服务项目建设逻辑
- 禁止背景化散文写法
- 禁止营销 / 宣传 / 产品介绍风格
- 段落必须可被审查与摘录

---

## 语气与风格锁定

- 使用政府 / 公安 / 信息化建设语境
- 语气客观、中性、非宣传
- 不得使用情绪化表达
- 不得使用模型自我描述
- 不得出现“建议性闲聊语气”

---

## 信息密度要求

- 删除无功能语句
- 优先结构清晰而非修辞
- 每段必须承担明确文档功能
- 禁止冗余解释

---

## 文件操作纪律

- 写作前必须假定文件已存在
- 若需修改 drafts/，只做局部补充
- 不得重建用户已有结构
- 不得输出与任务无关章节

---

## 图件生成执行顺序（强制）

1. 先完成正文结构生成
2. 根据 chart_registry.md 判断图件触发
3. 生成 drafts/chart/ 推理图
4. 根据图件类型校验一致性：
- 现状类 → 校验 specs/需求.md
- 建设类 → 校验 investment_schema.md
- 对比类 → 校验两者
5. 生成 output/chart/ HTML 源图
6. 导出 PNG
7. 最后进行正文图件引用

不得在正文未完成前提前生成正式图件。

## 投资预算生成规则（强制）

投资预算相关内容必须服从：

- specs/investment_schema.md
- specs/technical_baseline.md
- specs/需求.md
- 单价属于政策合规参数，而非估算参数
- 不得基于市场均值、经验值或模型推理生成


禁止模型自行设计预算结构或计算规则。


---

## 图件与技术示意生成规则（强制）

### 一、图件地位

- 图件属于正式方案交付物组成部分  
- 图件用于表达系统结构、部署关系与技术决策  
- 图件不得作为装饰性内容生成  
- 禁止生成与项目无关的示意图  

---

### 二、UI类图件控制规则（新增）

UI图件属于结构表达资产，
不属于技术路线表达资产。

UI图件必须：

- 不引入 specs 未定义功能
- 不展示未来扩展能力
- 不使用品牌UI风格
- 不生成营销式界面
- 不生成可运行前端代码

UI截图必须单独存放于：

output/screenshot/

不得混入 chart 目录。

---


### 三、图件生成位置纪律

- 推理或草稿图件 → drafts/chart/
- 正式交付图件 → output/chart/

禁止：

- 将 drafts 图件直接作为交付物
- 在 output 中生成未经推理的临时图
- 在正文中使用 ASCII 图或伪图替代专业图件

---

### 四、图件一致性约束

图件必须根据类型匹配事实来源：

- 现状类图件 → 必须与 specs/需求.md 一致
- 建设类图件 → 必须与 specs/investment_schema.md 一致
- 对比类图件 → 必须同时与两者一致

不得跨层引用事实来源。

---

### 五、图件表达原则

图件仅用于：

- 表达既定技术决策
- 解释系统结构关系
- 支撑方案评审理解

图件不得承担：

- 技术探索功能
- 方案创新展示功能
- 产品宣传功能

---

### 六、图件格式纪律

图件源格式唯一规则（不可违反）

- 正式图件的源格式必须为 HTML
- PNG / SVG 仅允许作为派生交付格式
- 不得将 PNG / SVG 作为源图件生成

Mermaid 仅允许用于 drafts 阶段推理图件。

禁止：

- 在未指定格式时自行选择复杂绘图方案
- 在正文 Markdown 中嵌入图形脚本
- 输出不可直接交付的中间格式

---

### 七、图件生成行为限制

- 图件生成不得改变方案文本结构
- 图件生成不得触发章节重写
- 图件生成不得扩展系统能力描述
- 图件仅表达，不得推导新设计
- 件生成不得规避 chart_registry 中的类别控制

---

### 八、图件属性定义

图件属于 **技术决策表达资产**，  
而非 **内容生成资产**。

---

### 九、图件输出与交付格式规则（强制）

1. 所有正式图件必须首先生成 HTML 版本  
   输出路径：output/chart/

2. HTML 图件属于权威源图件，不得跳过生成

3. PNG / SVG 图片属于派生交付资产  
   必须基于 HTML 图件生成

4. 图片生成方式：

   - 通过 HTML 渲染后导出
   - 或通过截图生成

5. 禁止：

   - 模型直接生成 PNG 作为源图
   - 跳过 HTML 图件直接生成图片
   - 图文内容不一致

6. 图片用途限定：

   - Word / PDF 嵌入
   - 招投标文件
   - 静态汇报材料

7. HTML 图件用途限定：

   - 技术讲解
   - 汇报演示
   - 结构审查

---

### 十、图件分辨率与比例纪律（强制）

- PNG 图片分辨率不得低于 1920×1080
- 禁止输出低清晰度图件
- 图件比例必须适用于 Word / PPT 嵌入
- 禁止生成不可读或文字模糊的图件

---


## 配置与图件权威来源规则（强制）

本 Skill 在生成以下内容时，必须服从权威配置模型文件：

权威文件：
specs/investment_schema.md

该文件为本项目资源配置、设备规模及部署结构的唯一事实来源。

一、适用范围

以下内容不得由模型推理生成，必须来源于 investment_schema.md：

服务器数量

设备配置参数

存储容量规模

角色划分数量

预算设备清单

部署结构图

网络拓扑图

数据流转图

二、禁止行为

禁止模型：

自行计算服务器数量

自行修改配置规模

自行扩展硬件类别

自行生成与配置模型不一致的图件

基于经验或常识推导容量与性能规模

三、图件生成约束

如方案章节触发图件生成需求：

A-建设类图件必须可由 investment_schema 推导。
A-现状类图件必须可由 specs/需求.md 推导。

图件中不得出现未在配置模型中定义的节点或组件

图件属于配置模型的可视化表达，而非架构再设计

四、冲突裁决规则

如方案正文与 investment_schema.md 存在冲突：

必须服从 investment_schema.md

不得以正文描述为准修改配置模型。

五、工程原则

investment_schema.md 属于工程事实层资产，
而非方案讨论层内容。

模型仅负责表达，不负责改写。


六、需求与配置冲突裁决规则（强制）

如 specs/需求.md 与 investment_schema.md 存在资源规模、设备数量或配置策略冲突：

必须服从 investment_schema.md

不得依据需求文本推导硬件规模调整

需求文件仅定义业务与功能边界，不裁决资源模型

资源配置权属于投资模型文件。
