# 多版本投标文件生成规则（强制）

## 一、场景定义

同一招标项目需生成多份（通常 3 份）不同风格的投标文件，分别对应不同投标主体。

---

## 二、项目隔离规则（不可违反）

### 目录结构

每个投标主体独立一个项目目录，共享招标侧信息，隔离投标侧信息：

```
bidding-skill/
├── shared/                          # 招标侧信息（三份共享，只读）
│   ├── 招标文件摘要.md
│   ├── 评分标准.md
│   ├── 技术要求.md
│   ├── 商务要求.md
│   └── 资格要求.md
├── projects/
│   ├── A/                           # 投标主体 A
│   │   ├── specs/                   # A 的投标侧信息
│   │   │   ├── 公司资料.md
│   │   │   ├── 报价清单.md
│   │   │   ├── 偏离表.md
│   │   │   └── style_profile.md     # ★ A 的写作风格档案
│   │   ├── output/
│   │   └── drafts/
│   ├── B/                           # 投标主体 B
│   │   ├── specs/
│   │   ├── output/
│   │   └── drafts/
│   └── C/                           # 投标主体 C
│       ├── specs/
│       ├── output/
│       └── drafts/
```

### 隔离约束

- 生成 A 的投标文件时，Claude **不得读取** B、C 的 specs/ 或 output/
- 生成 B 时，不得读取 A、C
- 三个项目目录之间 **零数据流动**
- 同一次对话中 **只处理一个投标主体**，切换主体前必须提醒用户

---

## 三、风格差异化规则（强制）

### 风格档案 style_profile.md

每个投标主体必须配置独立的写作风格档案，控制以下维度：

| 维度 | 说明 | 示例差异 |
|------|------|---------|
| 章节编排策略 | 在招标文件允许范围内调整详略 | A 重架构论述，B 重实施细节，C 重案例佐证 |
| 句式风格 | 句子结构与段落组织 | A 长句复合结构，B 短句清单化，C 中等句式+表格 |
| 论述切入角度 | 同一技术点的表达侧重 | A 从需求出发推导方案，B 从方案出发映射需求，C 从对比出发说明优势 |
| 图件风格 | 架构图配色/布局/标注方式 | A 蓝色系横向布局，B 绿色系纵向布局，C 灰色系矩阵布局 |
| 表格使用密度 | 正文中表格vs段落的比例 | A 偏文字段落，B 偏表格呈现，C 混合使用 |
| 强调手法 | 突出关键信息的方式 | A 加粗关键词，B 独立小节标题，C 对照表 |

### 风格差异度要求

- 任意两份投标文件中，**同一章节不得出现连续 3 句以上相同表述**
- 章节内部段落排列顺序必须存在可感知差异
- 技术方案图件必须使用不同的配色方案与布局方式
- 对同一招标要求的响应，必须使用不同的论述结构

---

## 四、防串标规则（强制 — 红线）

### 文档内容检查

- 每份文件中 **不得出现** 其他投标主体的公司名称、联系方式、人员姓名
- 不得出现其他投标主体的项目业绩或资质信息
- 不得出现暗示多份文件来源相同的表述

### 文档元数据检查

- Word 文件属性中的作者、公司、最后修改人必须与对应投标主体一致
- 文件创建时间、修改时间应有合理差异
- 文件名不得包含版本序号（如 A版/B版/1号/2号）

### 报价差异要求

- 三份报价 **不得相同**
- 分项报价结构允许存在合理差异（如同一设备不同品牌型号、不同配置方案）
- 报价偏差应在合理商业范围内，不得呈现等差/等比等人为规律

### 偏离差异要求

- 三份偏离表不必完全相同
- 允许不同投标主体在非实质性条款上有不同的偏离选择

---

## 五、交叉检查规则（生成完成后执行）

三份文件全部生成完成后，必须执行交叉检查，输出至各项目的 output/交叉检查报告.md：

### 文本相似度检查

- 逐章比对三份技术方案，标记相似度过高的段落
- 相似度阈值：同一章节任意两份之间 **逐段文本相似度不得超过 40%**

### 结构相似度检查

- 三份文件的章节编排顺序是否存在完全相同的模式
- 表格列数、行数、标题是否完全一致
- 图件布局、配色、标注位置是否可区分

### 数据泄露检查

- A 的文件中搜索 B、C 的公司名/人名/电话/邮箱
- B 的文件中搜索 A、C 的信息
- C 的文件中搜索 A、B 的信息

### 报价模式检查

- 三份总价是否呈现等差/等比/整数倍关系
- 分项报价是否存在大量完全相同的单价

---

## 六、构建顺序纪律

建议构建顺序：

1. 填写 shared/ 招标侧信息（一次填写，三份共用）
2. 分别填写 A/B/C 的 specs/（公司资料、报价、偏离、风格档案）
3. 逐个生成（A → 完成 → B → 完成 → C → 完成）
4. 三份全部完成后执行交叉检查
5. 针对交叉检查发现的问题进行差异化修改

**禁止：** 同时生成多份，这会增加模型在上下文中交叉污染的风险。

---

## 七、Word 构建差异化

build_docx.py 需支持以下差异化参数：

- 封面样式（字体布局、装饰线位置）
- 页眉页脚内容（投标主体名称、项目编号）
- 表格底色方案
- 正文行距微调（如 A 用 28 磅，B 用 29 磅，C 用 27 磅）
