风格名称：中标优先型 / 架构与落地平衡型

整体写作目标：
- 服务评分，而不是展示技术
- 提升专家理解效率
- 强调成熟度、稳定性、可控性
- 避免实验性、探索性语气

整体写作风格：
- 专业、克制、偏政务语境
- 论述完整但不冗长
- 关键评分点必须可见
- 避免营销化与宣传口吻
- 句式以中等句长为主，穿插结构化短句

章节展开策略：
- 占分高的评分项 → 必须显著展开
- 专家敏感区域（可靠性 / 风险 / 兼容 / 扩展） → 主动强化
- 非评分重点 → 简洁、干净、不过度铺陈

论述侧重点：
- 从“需求必然性”推导方案，而非直接给方案
- 强调系统合理性、必要性、稳定性
- 强调长期运行与管理可持续性
- 强调与既有系统/网络/设备兼容

技术表达策略：
- 禁止炫技式描述
- 禁止出现研发/探索/实验性措辞
- 优先使用工程确定性语言（如“采用成熟机制”“满足稳定运行要求”）
- 多使用“系统级能力”，少使用算法细节

风险控制强化策略（关键）：
必须显式体现：

- 稳定性保障机制
- 异常场景处理逻辑
- 扩展与演进兼容性
- 对既有环境影响控制
- 运维与管理友好性

专家阅读心理优化：
- 避免让专家推理系统结构
- 避免大段抽象描述
- 每个关键能力必须“可定位”
- 减少不必要的新概念

图件策略（中标向最重要模块）：

图件抽象层级：中高（既专业又清晰）

图件类型优先级：
1. 系统总体架构图（优先）
2. 部署结构图（高权重）
3. 网络拓扑图（简洁表达）
4. 数据流转图（辅助）

图件设计原则：
- 强调“体系完整”而非“设备堆叠”
- 强调层级与职责，而非硬件细节
- 强调逻辑清晰、阅读无负担
- 禁止复杂工程拓扑风

布局策略：
- 横向结构优先（政务/平台方案典型视觉）
- 中心化 / 平台化表达
- 节点分组而非设备罗列

节点命名策略：
- 业务语义优先（感知层 / 处理层 / 管控层）
- 避免厂商或型号主导图件
- 避免算法名词作为核心节点

视觉策略：
- 政务风 / 克制 / 简洁
- 不强调炫彩
- 不采用互联网产品视觉风格

图文关系规则：
- 图件用于降低理解成本
- 图件必须可直接支持评分点
- 正文不得重复解释图件显而易见关系

评分导向增强策略（核心竞争力）：
- 每个高分评分点必须在图件中可映射
- 图件结构服务评分逻辑，而非技术逻辑
- 专家可快速判断“方案成熟度”

禁止行为：
- 禁止展示探索性架构
- 禁止过度创新表达
- 禁止制造复杂系统印象
