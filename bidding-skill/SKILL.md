# BIDDING SKILL

## 权威输入域模型（不可违反）

shared/ 为唯一招标侧权威输入域：

1. shared/招标文件摘要.md
2. shared/评分标准.md
3. shared/技术要求.md
4. shared/商务要求.md
5. shared/资格要求.md
6. shared/废标条款.md

projects/A|B|C/specs/ 为唯一投标侧权威输入域：

- 公司资料.md
- 报价清单.md
- 偏离表.md
- style_profile.md

shared/ 属于只读权威域
投标生成阶段禁止修改 shared/
---

## 核心工作流程

```

招标文件(PDF/Word)
      ↓
[人工/辅助] 解析为 shared/ 结构化文件（唯一权威域）
      ↓
Claude 基于 shared/ + projects/X/specs/ 生成 output/
       ↓
  output/ → build_docx.py → Word 投标文件
```

---

## 招标文件解析规则（辅助环节）

当用户提供招标文件原文时：

- 必须提取至 shared/ 目录
- shared/ 不允许缺失
- 若 shared/ 不存在 → 必须先创建


提取行为约束：
- 仅提取，不推理、不补充、不扩展
- 提取结果必须经用户审核确认后方可作为权威输入
- 未经确认的提取结果存放于 drafts/，不得直接用于生成

提取目标文件：
- shared/招标文件摘要.md — 项目概况、采购方式、投标截止时间、开标时间
- shared/评分标准.md — 评分细则（技术分、商务分、价格分权重与子项）
- shared/技术要求.md — 技术参数、功能需求、性能指标（含★标参数）
- shared/商务要求.md — 交付周期、验收条件、付款方式、售后服务、质保要求
- shared/资格要求.md — 资质证书、业绩要求、人员要求、联合体限制

---

## 废标红线检查规则（强制）

生成投标文件前后必须逐条核查以下废标条款：

- 是否响应了全部带 ★ 的实质性技术参数
- 报价是否超过预算控制价（若招标文件公布）
- 投标文件格式是否符合招标文件要求（页码、目录、签章位置）
- 负偏离项是否在允许范围内
- 是否存在串标/围标嫌疑表述（如与其他供应商相同的笔误、联系方式等）
- 法人代表签字/授权书页是否保留占位

核查结果输出至 output/废标风险检查.md。

提取目标文件必须包含：

- shared/招标文件摘要.md
- shared/评分标准.md
- shared/技术要求.md
- shared/商务要求.md
- shared/资格要求.md
- shared/废标条款.md

---

## 评分导向写作规则（强制）

评分标准直接决定章节写作深度与篇幅分配：

- 占分 ≥15 的评分项 → 对应章节为写作重心，必须充分展开
- 占分 5-14 的评分项 → 对应章节中等篇幅，清晰响应
- 占分 <5 的评分项 → 对应章节简洁明确，不过度展开
- 无评分对应的章节 → 仅满足格式要求，不刻意增加篇幅

---

## 技术方案写作规则

- 技术方案必须逐条响应 shared/技术要求.md

- 响应方式：先引述招标要求原文，再阐述投标方案
- 带 ★ 参数必须正面响应，不得回避或模糊处理
- 偏离项唯一来源：projects/X/specs/偏离表.md
- 不得编造技术参数、性能数据或测试结果
- 不得引入招标要求之外的系统能力（除非评分标准明确鼓励）
- 图件生成规则与 proposal-writing-skill 一致（HTML 源 → PNG 派生）

---

## 商务方案写作规则

- 商务方案必须逐条响应 shared/商务要求.md
- 交付周期、质保期限等承诺必须与 projects/X/specs/偏离表.md 一致
- 售后服务方案需明确响应时间、服务方式、人员配备
- 不得超出公司实际能力进行承诺

---

## 报价文件写作规则（强制）

- 报价数据唯一来源：projects/X/specs/报价清单.md
- 不得基于市场均值、经验值或模型推理修改报价
- 报价格式必须与招标文件要求的格式完全一致
- 报价总价 = 各分项之和，必须算术校验
- 若招标文件公布预算控制价，投标总价不得超过

---

## 公司资料引用规则

- 公司信息唯一来源：projects/X/specs/公司资料.md
- 不得编造公司业绩、资质或人员信息
- 资质证书有效期必须覆盖投标截止日
- 项目业绩引用必须与合同/验收报告可对应

---

## 写作行为约束（强制规则）

- 不得整体重写既有章节
- 不得改变招标文件规定的章节标题与编号结构
- 若章节已存在，仅允许增量补充
- 禁止删除用户未明确要求删除的内容
- 禁止解释写作理由或模型决策过程
- 禁止营销/宣传/产品介绍风格（投标语境要求积极响应但不夸大）

---

## 语气与风格锁定

- 使用投标文件正式语境
- 语气：积极响应、专业严谨、不夸大
- 禁止情绪化表达
- 禁止模型自我描述
- 禁止"建议性闲聊语气"
- 对招标要求的响应使用"我方""投标人"等第一人称

---

## 文件操作纪律

- 写作前必须假定文件已存在
- 若需修改 drafts/，只做局部补充
- 不得重建用户已有结构
- 不得输出与任务无关章节

---

## 图件规则

与 proposal-writing-skill 一致：

- 正式图件源格式必须为 HTML → output/chart/
- PNG/SVG 为派生交付格式
- 图件必须与权威输入域严格一致（shared/ + projects/X/specs/）
- 不得引入任何权威文件未定义的系统元素

## 图件与技术示意生成规则（强制）

### 一、图件地位

- 图件属于正式方案交付物组成部分  
- 图件用于表达系统结构、部署关系与技术决策  
- 图件不得作为装饰性内容生成  
- 禁止生成与项目无关的示意图  

---

### 二、图件触发条件

当且仅当 specs/（按权威优先级）已提供足够结构化信息时，
或在用户显式要求生成图件时，允许触发图件生成，
但仍不得引入 specs 未定义元素，
才必须生成对应图件。

- 涉及网络结构、设备连接、链路关系 → 必须生成 **网络拓扑图**
- 涉及系统层级、组件构成、模块关系 → 必须生成 **系统架构图**
- 涉及数据流转、数据存储、交互路径 → 必须生成 **数据流转图**
- 涉及系统部署位置、节点分布、中心结构 → 必须生成 **部署结构图**

不得省略，不得替代。

若 specs 不足以支持图件生成：

- 禁止基于常识或行业惯例补充结构
- 必须输出 output/图件输入缺口清单.md

---

### 三、图件生成位置纪律

- 推理或草稿图件 → drafts/chart/
- 正式交付图件 → output/chart/

禁止：

- 将 drafts 图件直接作为交付物
- 在 output 中生成未经推理的临时图
- 在正文中使用 ASCII 图或伪图替代专业图件

---

### 四、图件一致性约束

所有图件必须：

- 与权威优先级中定义的 specs/ 文件严格一致
- 不得引入 specs 未定义的系统元素
- 不得扩展未裁决的技术组件
- 不得隐含新增系统能力
- 命名符合政府 / 公安方案语境
- 图件不得基于常识、行业惯例或模型推理补充 specs 未明确的元素
- 正文技术描述不得与既有图件结构发生冲突


若正文与图件存在不一致：

- 必须输出 output/图文一致性检查.md
- 修正其中一方以回归 specs 定义
- 不得直接以任一方为准


禁止：

- 自行设计新系统结构
- 引入未说明的第三方系统
- 使用厂商宣传式架构表达

---

### 五、图件表达原则

图件仅用于：

- 表达既定技术决策
- 解释系统结构关系
- 支撑方案评审理解

图件不得承担：

- 技术探索功能
- 方案创新展示功能
- 产品宣传功能

---

### 六、图件格式纪律

图件源格式唯一规则（不可违反）

- 正式图件的源格式必须为 HTML
- PNG / SVG 仅允许作为派生交付格式
- 不得将 PNG / SVG 作为源图件生成

Mermaid 仅允许用于 drafts 阶段推理图件。

禁止：

- 在未指定格式时自行选择复杂绘图方案
- 在正文 Markdown 中嵌入图形脚本
- 输出不可直接交付的中间格式

---

### 七、图件生成行为限制

- 图件生成不得改变方案文本结构
- 图件生成不得触发章节重写
- 图件生成不得扩展系统能力描述
- 图件仅表达，不得推导新设计

---

### 八、图件属性定义

图件属于 **技术决策表达资产**，  
而非 **内容生成资产**。

---

### 九、图件输出与交付格式规则（强制）

1. 所有正式图件必须首先生成 HTML 版本  
   输出路径：output/chart/

2. HTML 图件属于权威源图件，不得跳过生成

3. PNG / SVG 图片属于派生交付资产  
   必须基于 HTML 图件生成

4. 图片生成方式：

   - 通过 HTML 渲染后导出
   - 或通过截图生成

5. 禁止：

   - 模型直接生成 PNG 作为源图
   - 跳过 HTML 图件直接生成图片
   - 图文内容不一致

6. 图片用途限定：

   - Word / PDF 嵌入
   - 招投标文件
   - 静态汇报材料

7. HTML 图件用途限定：

   - 技术讲解
   - 汇报演示
   - 结构审查

---

### 十、图件分辨率与比例纪律（强制）

- PNG 图片分辨率不得低于 1920×1080
- 禁止输出低清晰度图件
- 图件比例必须适用于 Word / PPT 嵌入
- 禁止生成不可读或文字模糊的图件

---

### 图件复杂度控制规则（强制）

- 图件必须以“专家快速理解”为优先目标
- 禁止无必要的节点扩张
- 禁止展示与评分无关的技术细节
- 单张图件不得形成阅读负担
- 优先体现层级、职责与关系，而非设备堆叠

---

### 图件默认策略（强制）

- 在不存在明确复杂结构要求时，优先采用简化表达
- 优先减少视觉噪音
- 优先保障专家阅读效率

---

## shared/ 权威文件完整性检查规则（强制）

在生成任何 output/ 文件前必须验证：

【文件完整性检查】

以下文件必须存在且非空：

- shared/招标文件摘要.md
- shared/评分标准.md
- shared/技术要求.md
- shared/商务要求.md
- shared/资格要求.md
- shared/废标条款.md

【结构有效性检查】

以下结构必须可识别：

- shared/技术要求.md 中带 ★ 的条款
- shared/评分标准.md 中的分值与权重结构

【禁止行为】

- 任一文件缺失 → 禁止生成 output/
- 文件存在但内容为空 → 视为缺失
- 文件结构明显不完整 → 必须停止并提示用户

检查失败输出：

output/shared完整性检查.md
