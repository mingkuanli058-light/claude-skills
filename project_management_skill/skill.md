# PROJECT MANAGEMENT SKILL

本 Skill 用于在 Claude Code 中执行项目管理全生命周期作业。

Skill 的职责是控制项目管理文档的生成逻辑、结构稳定性与合同映射一致性，
而非提供项目管理咨询、方法论讲解或实施策略。

所有项目管理文档被视为：

合同责任与交付义务的结构化映射，而非工作建议书。

---

## 强制基线输入（Required Specs）

执行任何模块前必须存在：

- specs/合同约束.md
- specs/范围基线.md
- specs/技术基线.md
- specs/商务约束基线.md
- specs/责任矩阵.md
- specs/验收判定矩阵.md

若缺失任一文件，必须先生成或确认。

---

## 权威优先级（不可违反）

1. 合同约束基线  
   → 定义法律履约义务、付款条件、违约责任。  
   → 一切结构不得违反合同条款。

2. 范围基线  
   → 定义项目交付边界（In Scope / Out of Scope）。  
   → WBS 仅可围绕范围基线展开。

3. 技术基线  
   → 定义最低技术能力与性能指标。  
   → 禁止降低指标，禁止扩展合同外能力。

4. 验收判定矩阵  
   → 定义可测试、可裁决的验收标准。  
   → WBS 末级节点必须可映射至验收判定项。

5. 责任矩阵  
   → 定义甲乙双方及第三方责任边界。  
   → 进度偏差与履约争议必须可归责。

6. 商务约束基线（内部）  
   → 定义利润、成本、现金流与经营红线。  
   → 不得为技术优化突破商务约束。

## 【SCOPE 模块】范围管理

### WBS 本质规则（最高优先级）

WBS 的拆解对象是：

✔ 可交付成果
✔ 合同义务
✔ 验收单元
✔ 管控节点

禁止拆解为：

✘ 日常动作
✘ 工作习惯
✘ 技术实现步骤
✘ 部门内部流程

WBS 必须围绕结果与责任，而非行为。

### 合同映射约束（强制）

WBS 必须可被以下要素反向解释：

- 合同条款
- 建设范围
- 验收依据
- 付款节点
- 责任主体

任何无法映射至合同责任的 WBS 节点均视为非法节点。

禁止生成：

✘ 合同未定义工作
✘ 推测性任务
✘ 常识性补充项

### 分解行为约束（强制规则）

- 不得改变既有 WBS 层级结构
- 不得重排节点逻辑顺序
- 不得合并用户未指示合并的节点
- 不得删除既有节点
- 若节点存在，仅允许增量细化

WBS 结构属于冻结资产，而非可重写文本。

### WBS 结构纪律

WBS 必须满足：

✔ 层级清晰（Level 1 / Level 2 / Level 3）
✔ 编号稳定且不可漂移
✔ 同层级粒度一致
✔ 不混合交付物与动作描述

禁止：

✘ 层级跳跃
✘ 语义重复
✘ 不可验收节点
✘ 管理项与实施项混杂

### 验收导向规则（核心规则）

每一个 WBS 末级节点必须隐含：

- 完成判定条件
- 验收依据
- 输出物或状态变更

无法定义完成状态的节点不得进入 WBS。

### 商务 / 管控节点强制存在规则

WBS 必须显式包含：

✔ 项目管理类节点
✔ 商务与合规类节点
✔ 技术实施类节点
✔ 验收与移交类节点

禁止仅生成技术任务型 WBS。

### 技术路径污染禁止规则

WBS 不得体现：

✘ 算法实现逻辑
✘ 产品配置细节
✘ 编码行为
✘ 工程实施步骤

WBS 是管理结构，而非技术设计书。

## 【SCHEDULE 模块】进度管理

### 里程碑定义规则

里程碑必须满足：

✔ 与 WBS 一级或二级节点对应
✔ 具备明确验收事件（非活动完成）
✔ 可关联付款节点或合同节点
✔ 日期可被合同条款约束

禁止生成：

✘ 无验收依据的里程碑
✘ 纯技术状态里程碑（如"开发完成"）
✘ 模糊时间节点（如"视情况确定"）

### 进度报告规则

进度报告必须包含：

✔ 当前完成状态（对比基线）
✔ 偏差分析（延误原因可归责性判定）
✔ 预测完成时间
✔ 风险预警（若存在进度风险）

禁止：

✘ 描述工作过程（仅报告结果状态）
✘ 使用进展百分比（除非有客观计量依据）

---

## 【RISK 模块】风险管理

### 风险登记规则

风险条目必须包含：

✔ 风险编号（稳定不漂移）
✔ 风险描述（可量化的不确定性，非已发生问题）
✔ 触发条件
✔ 影响范围（范围 / 进度 / 成本 / 质量）
✔ 责任方（甲方 / 乙方 / 第三方）
✔ 应对策略（接受 / 规避 / 转移 / 缓解）

禁止：

✘ 将已发生问题登记为风险
✘ 登记无控制手段的宿命性风险
✘ 仅描述风险而无应对策略

### 风险分级规则

风险按影响等级分为：

- 高：可能导致合同违约、验收失败或付款延误
- 中：可能导致里程碑偏移或质量降级
- 低：局部影响，可在项目内自行消化

---

## 【CHANGE 模块】变更管理

### 变更识别规则

以下情形强制触发变更流程：

✔ 新增系统能力
✔ 新增接口责任
✔ 新增部署范围
✔ 新增服务责任
✔ 修改验收标准
✔ 修改交付时间

禁止：

✘ 未经审批擅自执行变更工作
✘ 将变更内容纳入现有 WBS 节点
✘ 以"优化"或"完善"名义规避变更流程

### 变更申请要素

变更申请必须包含：

✔ 变更描述（原条款对比新要求）
✔ 变更原因（客观事实，非主观需求）
✔ 影响分析（范围 / 进度 / 成本 / 质量）
✔ 处置建议（含拒绝理由或批准条件）

---

## 【ACCEPT 模块】验收管理

### 验收范围限定

验收对象仅限于：

✔ 合同约定内容
✔ 范围基线包含范围
✔ 经审批变更内容

验收不得引入新需求、新标准、新系统能力。

### 验收文档规则

验收方案必须包含：

✔ 验收依据（合同条款 + 技术规范）
✔ 验收范围（对应 WBS 末级节点）
✔ 验收判定标准（客观可测量）
✔ 验收主体责任（甲乙双方）

验收报告必须包含：

✔ 验收结论（通过 / 有条件通过 / 拒绝）
✔ 验收缺陷清单（若存在）
✔ 整改时限要求（若适用）

---

## 【CONTRACT 模块】合同履约管理

### 付款节点跟踪规则

付款节点必须与 WBS 验收节点一一对应。

付款节点记录必须包含：

✔ 合同付款条款编号
✔ 触发条件（验收事件）
✔ 应付金额（含税）
✔ 计划触发日期
✔ 实际触发状态

### 履约风险预警规则

以下情形触发履约风险预警：

- 里程碑偏移超过合同容忍期
- 验收被拒且整改期不足
- 甲方配合事项逾期未完成
- 第三方不配合影响主合同履约

---

## 通用写作纪律（跨模块强制）

- 使用工程管理语境
- 描述必须为可交付物或状态语义
- 不使用口语化动作词（如：开发、对接、处理等）
- 不使用模糊状态词（如：优化、完善、支持等）
- 所有文档内容必须具备可审计语义

---

## 文件操作纪律

- 写作前必须假定文件已存在
- 若修改 drafts/，只允许局部细化
- 不得整体重建已有文档
- 不得跨层级扩展结构

---

## 默认目标

生成内容必须可直接用于：

✔ 项目实施计划
✔ 合同履约管理
✔ 进度管控
✔ 验收管理
✔ 付款节点控制
✔ 风险防控
✔ 变更管理

---

## 禁止行为（绝对禁止）

- 禁止模型自行设计项目范围
- 禁止基于行业惯例补充工作
- 禁止解释项目管理理论
- 禁止输出建议性内容
- 禁止改变文档语义体系
- 禁止跨模块混用规则

---

## 基线冲突裁决规则

若生成内容与六项基线存在语义冲突：

1. 优先判定是否违反合同约束基线；
2. 若不违反合同但违反范围基线，视为范围扩展；
3. 若违反技术基线，视为技术降级或扩展；
4. 若违反验收判定矩阵，视为不可验收节点；
5. 若违反责任矩阵，视为责任漂移；
6. 若违反商务约束，视为内部经营风险。

任何冲突必须明确指出并拒绝生成非法结构。
