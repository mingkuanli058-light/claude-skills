## 一、候选技术方案集合

> 基线：有效可用容量 ≥ 700TB，日增 0.6TB，在线周期 ≥ 3 年，并发 ≥ 20 路，公安专网部署，原始证据数据不可改变

---

### 方案一：集中式 SAN 存储方案

**架构思路**

- 采用企业级光纤 SAN（FC-SAN）存储阵列作为核心存储资源池
- 前端部署应用服务器通过 FC 交换机连接存储阵列
- 存储阵列采用 RAID6 或同等冗余保护机制，扣除保护开销后提供 ≥ 700TB 有效容量
- 管理平台通过标准文件接口或块设备接口完成数据写入与检索
- 分时上传调度模块部署于应用服务器层，实现削峰控制

**适用条件**

- 数据访问模型以顺序写入、低并发调阅为主（与本项目吻合）
- 运维团队熟悉传统存储架构，缺少分布式运维经验
- 项目周期紧张，需快速上线交付
- 评审方倾向于成熟稳定、风险最低的技术路线

**主要风险**

| 风险项 | 说明 |
|--------|------|
| 纵向扩展瓶颈 | 单阵列扩展到一定规模后需增加阵列，管理复杂度上升 |
| 单位容量成本偏高 | 相较分布式方案，FC-SAN 的 TB 单价明显更高，投资合理性可能受到评审质疑 |
| 存储控制器单点风险 | 虽有双控冗余，但故障切换期间业务可能中断 |
| 3 年后扩容路径 | 当数据累积超出初始容量时，扩容需新增阵列及交换机端口 |

---

### 方案二：分布式存储集群方案

**架构思路**

- 采用分布式存储软件（如开源 Ceph 或国产商业分布式存储）构建统一存储资源池
- 多台标准化存储服务器组成集群，通过万兆以太网互联
- 数据采用多副本（建议 2 副本 + 纠删码混合策略）保护，扣除保护开销后提供 ≥ 700TB 有效容量
- 对外提供对象存储接口（S3 兼容）或文件接口供管理平台调用
- 集群原生支持横向扩展，新增节点即可扩充容量与性能

**适用条件**

- 预期数据量持续增长，3 年后有明确扩容需求
- 建设方希望控制 TB 单价，追求投资效率
- 运维团队具备或愿意培养分布式系统运维能力
- 评审方认可分布式存储在政务/公安领域的落地案例

**主要风险**

| 风险项 | 说明 |
|--------|------|
| 运维复杂度高 | 集群运维需专业能力，故障定位难度大于传统存储 |
| 集群最小规模要求 | 一般需 ≥ 3 节点起步，初始投入有门槛 |
| 网络依赖度高 | 节点间数据同步依赖内部网络，网络故障影响面大 |
| 开源方案的长期支持风险 | 若采用开源路线，需评估长期技术支持与版本升级保障 |

---

### 方案三：混合存储方案（SAN/NAS + 对象存储分层）

**架构思路**

- **在线存储层**：采用中端 SAN 或 NAS 存储阵列承载近期（如 1 年内）高频调阅数据
- **近线/归档层**：采用分布式对象存储或大容量 NAS 承载超过高频期的在线数据
- 管理平台内置数据生命周期策略，根据时间自动将数据从在线层迁移到近线层
- 原始证据数据在两层之间保持完整不变，迁移仅为存储介质变更
- 副本/归档副本可在归档层实施空间优化（符合需求边界）

**适用条件**

- 希望兼顾近期访问性能与大容量长期存储的经济性
- 项目预算允许适度增加架构复杂度以换取 TCO 优化
- 评审方关注投资合理性，需要体现分层存储的成本优势
- 存在明确的数据冷热分层需求（近期高频调阅 vs. 长期低频留存）

**主要风险**

| 风险项 | 说明 |
|--------|------|
| 架构复杂度增加 | 两层存储 + 生命周期策略增加系统集成与测试工作量 |
| 数据迁移一致性 | 在线层到近线层的自动迁移需确保证据完整性校验机制 |
| 运维双栈负担 | 运维团队需同时掌握两种存储体系 |
| 过度设计风险 | 若实际调阅行为无明显冷热分化，分层收益不显著，评审可能质疑必要性 |

---

### 方案四：Scale-Out NAS 集群方案

**架构思路**

- 采用横向扩展型 NAS（Scale-Out NAS）设备构建统一文件存储池
- 多个 NAS 节点通过集群协议互联，统一命名空间管理
- 通过标准 NFS/CIFS 文件接口对管理平台提供服务
- 数据保护采用节点级冗余 + RAID 双重保护
- 扩容时增加 NAS 节点，容量与性能线性增长

**适用条件**

- 管理平台以文件接口方式存取音视频数据（无需对象存储接口）
- 运维团队熟悉 NAS 管理，希望降低技术学习曲线
- 需要横向扩展能力但不希望引入分布式存储软件栈
- 评审方倾向设备级方案，偏好交钥匙式交付模式

**主要风险**

| 风险项 | 说明 |
|--------|------|
| 厂商锁定 | Scale-Out NAS 通常为品牌专有方案，扩容需同品牌设备 |
| 单位容量成本居中 | 低于 FC-SAN 但高于纯分布式方案 |
| 文件系统性能边界 | 当单一命名空间文件数极大时，元数据性能可能下降 |
| 产品生命周期风险 | 3-5 年后同系列产品是否延续直接影响扩容 |

### 逐维度说明

**1. 合规性与制度匹配**

- 方案一、四：设备形态清晰，采购标的明确，符合政府采购与公安信息化项目设备清单式评审习惯
- 方案二：分布式软件 + 通用服务器模式在部分评审中存在"软硬件拆分定价"争议
- 方案三：双层架构需在可研报告中额外论证分层必要性，增加评审解释成本

**2. 技术可行性**

- 方案一：FC-SAN 在公安音视频存储领域有大量部署案例，技术路线无争议
- 方案二：分布式存储已有公安行业落地，但 700TB 级别集群的运行稳定性需厂商案例支撑
- 方案三：两套存储体系集成 + 生命周期策略开发，系统集成验证工作量显著增加
- 方案四：Scale-Out NAS 在视频监控与音视频存储领域应用成熟，技术路线可行

**3. 实施风险**

- 方案一：交付模式成熟，实施周期短，风险最低
- 方案二：集群部署调优周期较长，对实施团队能力有要求
- 方案三：双栈集成 + 数据迁移策略开发，实施链条最长，风险最高
- 方案四：设备级交付，实施复杂度适中

**4. 运维与复杂度**

- 方案一：单一存储体系，运维模型简单，基层运维可承接
- 方案二：集群运维需专业能力，故障排查依赖厂商支持
- 方案三：需同时运维两套存储 + 迁移策略，运维负担最重
- 方案四：NAS 管理界面友好，扩展操作标准化，运维门槛适中

**5. 扩展与演进能力**

- 方案一：纵向扩展受阵列物理槽位限制，超出后需新增阵列，扩展路径不连续
- 方案二：原生横向扩展，增加节点即可线性扩充，扩展能力最强
- 方案三：近线层可横向扩展，在线层仍受设备限制，整体扩展能力居中偏上
- 方案四：节点扩展能力良好，但受限于单一厂商产品线延续性

**6. 投资与采购合理性**

- 方案一：FC-SAN 的 TB 单价在 700TB 规模下偏高，评审中投资合理性论证压力大
- 方案二：通用服务器 + 软件定义模式 TB 单价有优势，但需论证软件授权费合理性
- 方案三：两套存储体系总体投资可能低于纯 SAN，但架构复杂性带来的集成费用抵消部分成本优势
- 方案四：TB 单价居中，设备形态明确，采购定价透明，投资论证压力适中

**7. 与既有系统兼容性**

- 方案一、四：与既有系统并行部署无干扰，接口标准化，兼容性好
- 方案二、三：需确认管理平台对对象存储接口/多存储后端的适配情况

- 原始证据数据完整性校验机制（如哈希校验）为所有方案的共性实施要求

## 五、推荐结论

### 推荐方案：方案四（Scale-Out NAS 集群方案）

### 推荐依据

推荐结论基于**风险最小化**与**约束条件匹配度**，而非技术先进性：

1. **实施风险可控**：设备级交付模式成熟，实施链条短，项目按期交付概率高
2. **投资合理性可论证**：TB 单价居中，设备形态清晰，采购定价透明，无软件授权定价争议
3. **运维复杂度适中**：不要求分布式集群运维能力，基层运维团队可承接
4. **扩展能力满足需求**：横向扩展能力可覆盖 3 年在线周期内的容量增长
5. **评审逻辑通顺**：设备清单式采购、成熟技术路线、明确扩容路径，符合公安信息化项目评审惯例
6. **与既有系统无冲突**：标准文件接口，与原系统并行运行无干扰

### 备选方案

- 若评审方明确倾向最保守路线且预算充裕 → 回退至**方案一（集中 SAN）**
- 若预算压力大且运维能力有保障 → 可选择**方案二（分布式集群）**
- **方案三不推荐**作为首选：架构复杂度与本项目 20 路并发、存储调阅型的系统定位不匹配，存在过度设计风险
